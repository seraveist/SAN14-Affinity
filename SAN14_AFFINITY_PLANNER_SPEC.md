# 삼국지14 친애 편성기 기획/구현 정리

## 1. 프로젝트 개요

삼국지14의 장수 친애 관계, 능력치, 개성 데이터를 기반으로 장수 조합과 집단 편성을 추천하는 정적 웹 앱이다.

주요 목적은 두 가지다.

1. 특정 장수를 중심으로 가능한 친애 네트워크와 집단 후보를 확인한다.
2. 보유 장수 목록을 입력하면, 입력한 집단당 최대 인원 기준으로 효율적인 조합을 자동 편성한다.

앱은 서버 없이 동작하는 정적 웹 앱으로 구성한다. GitHub에 올린 뒤 GitHub Pages로 접속 가능하게 하고, GitHub Release는 버전 배포와 압축 파일 제공용으로 사용한다.

## 2. 사용 데이터

### 2.1 친애 관계 JSON

원본 파일:

```text
C:\Users\User\Downloads\affinity_index.json
```

주요 구조:

```text
metadata
characters
edges
indices
```

사용 필드:

```text
characters[].id
characters[].name
characters[].support_flag
edges[].source_id
edges[].source_name
edges[].relation
edges[].target_ids
```

관계 타입:

```text
☆ : 별
◎ : 쌍원
○ : 단원
X : 혐오
```

편성 로직에서는 `☆`, `◎`, `○`만 사용하고 `X`는 제외한다.

### 2.2 장수 능력치/개성 엑셀

원본 파일:

```text

```

주요 시트:

```text
반에디터_원본
```

`전체무장 능력치개성` 시트에서 사용하는 정보:

```text
무장번호
무장명
통솔
무력
지력
정치
매력
개성1~개성5
```

## 4. UX 기획

앱은 두 가지 모드를 제공한다.

```text
보유 장수 편성
중심 분석
```

### 4.1 보유 장수 편성 모드

사용자가 보유한 장수 이름을 여러 명 입력한다.

입력 예시:

```text
유비, 관우, 장비, 황충, 조운, 마초, 마대, 제갈량
```

편성 기준:

```text
목표 집단 수: 최종적으로 맞출 집단 갯수
집단당 최대 인원: 각 집단에 들어갈 수 있는 최대 인원
```

동명이인 처리:

```text
동명이인 후보 수보다 적게 입력된 이름은 선택 콤보박스를 표시한다.
콤보박스에는 이름 구분값과 통|무|지|정|매 요약을 함께 표시한다.
동명이인 이름이 후보 수만큼 입력되면 모든 후보가 존재한다고 보고 자동 확정한다.
확정된 동명이인은 장소(1), 장소(2)처럼 이름 뒤 번호로 구분한다.
```

출력:

```text
편성 결과 탭
1집단
- 구성원 배지(이름 + 버프점수)
- 작은 카드형 연결 근거

2집단
...

의형제 추천 탭
- 추천 의형제 세트
- 점수 상승 배지
- 실제 변동이 있는 편성 변화 배지
- 카드형 예상 편성 결과
```

목표 집단 수와 집단당 최대 인원은 동시에 적용한다.
집단당 최대 인원은 기본값 10명, 입력 상한 20명으로 제한한다.
예를 들어 35명을 4집단, 최대 8명으로 편성하는 것은 불가능하므로 최소 5집단으로 보정한다.

### 4.2 중심 분석 모드

특정 장수 한 명을 입력한다.

입력 예시:

```text
관우
```

출력:

```text
친애 목록
집단 추천
```

출력 형식 예:

```text
관우 ☆ 장비
관우 ◎ 조운
관우 ☆ 관평
관우 ☆ 장비 ◎ 조운
```

## 5. 핵심 점수 로직

### 5.1 친애 관계 점수

관계 강도는 다음처럼 계산한다.

```text
☆ = 3
◎ = 2
○ = 1
X = 제외
```

관계는 단방향/역방향/쌍방향 모두 탐색한다.

표시 방식:

```text
A -> B  : A가 B를 친애
A <- B  : B가 A를 친애
A <-> B : 양쪽 모두 친애
```

### 5.2 개성 점수

개성 점수는 셀 배경색 기준으로 계산한다.

```text
금색 개성: +5
파란색 개성: +2
빨간색 개성: +0
```

추가 보정:

```text
악주: +1
응원: +1
```

예시:

```text
응원은 파란색 개성이므로 기본 +2, 범위 버프 보정 +1, 총 +3
```

### 5.3 능력치 우선도

현재 장수 기본 전투/편성 가치는 다음 우선순위로 계산한다.

```text
1순위 primary   = max(통솔 + 무력, 통솔 + 지력) + 개성점수
2순위 secondary = 통솔 + 정치 + 개성점수
3순위 third     = 지력 + 정치 + 개성점수
4순위 fourth    = 통솔 + 개성점수
```

현재 UI에서는 주로 `primaryScore`를 표시한다.

## 6. 버프 중첩 로직

같은 집단 안에 있는 장수끼리 `☆`, `◎`, `○` 관계가 있으면 양쪽 모두 버프 1중첩을 얻는 것으로 계산한다.

예시:

```text
관우 <-> 장료
관우 <-> 서황
```

버프 중첩:

```text
관우 +2
장료 +1
서황 +1
```

집단별 편성 카드에는 구성원 배지와 연결 근거만 표시한다.
구성원 배지는 `관우 +2`처럼 이름과 버프점수만 표시한다.
연결 근거는 작은 카드로 표시하고, 첫 번째 기준 장수 순서대로 정렬한다.
의형제는 별도 `의형제 추천` 탭에서 자동 추천세트와 예상 재편성만 표시하고, 결혼 후보는 표시하지 않는다.

## 7. 집단별 의형제 후보

집단 카드 안에서 같은 집단의 3인 조합을 단순 나열하는 `의형제 후보` 섹션은 제공하지 않는다.
의형제 관련 출력은 보유 장수 전체를 대상으로 재계산하는 `의형제 추천` 탭에서만 제공한다.

## 8. 결혼 후보

결혼 후보와 미혼 가정 시뮬레이션은 출력하지 않는다.
결혼은 기본 집단 점수와 버프 중첩 계산에도 반영하지 않는다.

## 9. 보유 장수 편성 알고리즘

완전탐색은 보유 장수가 많아지면 경우의 수가 급격히 커진다. 따라서 현재 구현은 정적 웹 앱에서 빠르게 동작하도록 휴리스틱 방식으로 고정한다.

### 9.1 기본 흐름

```text
1. 입력된 장수명을 ID로 해석한다.
2. 동명이인은 선택 UI 또는 후보 수만큼의 반복 입력으로 ID를 확정한다.
3. 별(☆) 관계로 강하게 연결된 컴포넌트를 먼저 만든다.
4. 각 컴포넌트를 최대 집단 인원 이하로 유지한다.
5. 남은 장수를 기존 집단에 넣었을 때의 marginal gain을 계산한다.
6. 친애 연결, 관계 강도, 장수 점수, 개성 점수를 기준으로 배정한다.
7. 어떤 집단에도 충분히 어울리지 않으면 새 집단을 만든다.
8. 3인 이하 소집단이 남으면 교환/흡수/병합 재배치를 검토한다.
9. 점수가 상승하는 재배치만 적용한다.
10. 목표 집단 수와 집단당 최대 인원에 맞도록 병합/분할 후처리를 적용한다.
11. 각 집단을 점수순으로 정렬해 출력한다.
```

### 9.2 집단 점수

집단 점수는 다음 요소를 합산한다.

```text
장수 primaryScore 합
버프 중첩 수
친애 관계 강도
```

현재 구현상 대략적인 계산:

```text
집단점수 = primaryScore 합 + 버프중첩합 * 15 + 관계강도합 * 10
```

이 점수는 절대적인 게임 공식이 아니라 추천 정렬용 내부 지표다.

### 9.3 장수 추가 평가

기존 집단에 새 장수를 넣을 때는 다음을 본다.

```text
집단 내부 친애 연결 수
관계 강도
별 관계 여부
장수 primaryScore
개성 점수
```

즉, 단순히 능력치가 높은 장수보다 기존 집단과 실제로 연결되는 장수를 우선한다.

### 9.4 재배치 후처리

초기 편성 뒤 3인 이하 집단이 남으면 소집단으로 보고 다음 재배치를 추가로 탐색한다.

```text
1. 소집단은 3인, 2인, 1인 순서로 검토한다.
2. 포화 집단 교환, 빈자리 있는 큰 집단 흡수, 소집단끼리 병합 순서로 후보를 만든다.
3. 포화 집단에 소집단을 넣을 때는 같은 수의 외곽 장수를 분리한다.
4. 분리된 장수가 다른 소집단과 이어지면 즉시 병합 후보로 계산한다.
5. 1인 집단은 앵커 이동과 빈자리 보강 재배치도 추가로 검토한다.
6. 전체 점수가 상승하는 경우에만 재배치를 적용한다.
7. 적용 후 소집단 목록을 다시 계산하고, 더 이상 개선이 없을 때 종료한다.
```

### 9.5 집단 수/인원 후처리

기본 편성과 재배치 후 목표 집단 수와 집단당 최대 인원을 함께 맞춘다.

```text
1. 입력된 집단당 최대 인원을 기준으로 가능한 최소 집단 수를 계산한다.
2. 입력 목표가 불가능하면 가능한 범위로 보정한다.
3. 현재 집단 수가 목표보다 많으면 점수 손실이 가장 적은 병합 또는 분산 병합을 반복한다.
4. 현재 집단 수가 목표보다 적으면 점수 손실이 가장 적은 집단 분할을 반복한다.
5. 의형제 추천의 예상 편성도 같은 집단 수/인원 기준으로 다시 계산한다.
```

예시:

```text
유비, 공손찬을 노식 집단으로 이동
감씨, 미씨, 후씨, 위연을 노식 집단에 합류
진도, 등지를 관우/조운 집단의 빈자리에 보강

하후돈, 하후연을 조조 집단에 합류
조인, 조홍을 조조 집단에 합류
분리된 우금, 포신, 창희, 장패를 별도 집단으로 병합
```

### 9.6 의형제 추천

의형제 추천은 사용자가 직접 조합을 입력하는 방식이 아니라, 보유 장수 편성 결과를 기준으로 자동 계산한다.

```text
1. 현재 편성에서 3인 이하 소집단과 1인 집단을 찾는다.
2. 소집단 장수와 큰 집단의 앵커 장수를 포함하는 3인 의형제 후보를 만든다.
3. 각 후보 세트를 임시 ☆ 연결로 가정하고 편성을 다시 계산한다.
4. 총점 상승, 1인 집단 감소, 소집단 감소, 집단 수 감소를 함께 평가한다.
5. 점수만 오르고 재편성 효과가 약한 후보보다, 집단 재배치를 열어주는 후보를 우선한다.
6. 상위 추천세트와 예상 편성 변화를 `의형제 추천` 탭에 표시한다.
7. 점수 상승은 추천 카드 우상단 배지로 표시하고, 예상 변화에는 실제 변동이 있는 집단/1인 집단 배지만 표시한다.
8. 예상 편성은 집단명, 점수, 구성원 배지만 있는 카드형 목록으로 표시한다.
```

## 10. 중심 분석 알고리즘

중심 장수 분석은 단일 장수를 기준으로 최대 4단계까지 탐색한다.

기본 규칙:

```text
1. 중심 장수 A를 선택한다.
2. A가 친애하는 장수와 A를 친애하는 장수를 합쳐 전체 직접 친애 목록으로 표시한다.
3. B -> C -> D -> E 형태로 최대 4단계까지 탐색한다.
4. 정방향뿐 아니라 역방향 연결도 포함한다.
5. `집단 추천` 카드에 3인 이상 경로만 점수가 높은 순서로 출력한다.
6. 같은 경로 안에서 이미 방문한 장수는 재방문하지 않는다.
```

출력은 단계별 섹션으로 나누지 않고, `집단 추천` 카드 안에서 3인 이상 경로를 점수가 높은 순서로 최대 50개 표시한다.
`집단 추천` 카드의 연결 근거 목록은 프레임 안에 표시하고 세로 스크롤을 사용한다.
중심 분석 모드에서는 `편성 결과`와 `의형제 추천` 결과 탭을 표시하지 않는다.

## 11. 출력 형식

### 11.1 보유 장수 편성

```text
1집단 · 관우, 조운, 관평

구성:
관우 +7
조운 +5
관평 +5

연결 근거:
관우 ☆ 장비
관우 ◎ 조운
관평 ☆ 관은병
```

### 11.2 중심 분석

```text
친애 목록 · 관우

관우 ☆↔ 장비
```

## 12. 배포 방식

도메인은 붙이지 않는다.

접속용 배포는 GitHub Pages를 사용한다.

기본 URL 형태:

```text
https://{github-id}.github.io/{repo-name}/
```

권장 저장소 구조:

```text
repo/
├─ docs/
│  ├─ index.html
│  ├─ app.js
│  ├─ styles.css
│  └─ data.js
├─ tools/
│  └─ build_data.py
└─ README.md
```

현재 `webapp/` 폴더는 GitHub Pages용으로 `docs/`에 복사하거나 이름을 바꿔 사용할 수 있다.

GitHub Release는 버전 배포용으로 사용한다.

```text
v0.1.0
- san14-affinity-planner.zip
- 변경사항 요약
- 사용 방법
```

즉, 역할은 다음처럼 나눈다.

```text
GitHub Pages  : 바로 접속해서 쓰는 앱
GitHub Release: 특정 버전 다운로드/보관
```

## 14. 참고사항과 한계

### 14.1 동명이인

삼국지14 데이터에는 같은 이름의 장수가 존재한다.

현재 구현은 보유 장수 편성에서 동명이인 후보 수보다 적게 입력된 이름에 선택 UI를 표시한다.

후보 수만큼 같은 이름이 입력되면 모든 후보가 존재한다고 보고 자동 확정한다.

```text
순욱(1)
순욱(2)
```

### 14.2 완전 최적화가 아님

현재 보유 장수 편성은 완전탐색 최적해가 아니라 휴리스틱 추천이다.

이유:

```text
보유 장수 수가 많아질수록 집단 배치 경우의 수가 폭증한다.
정적 웹 앱에서 즉시 반응해야 한다.
사용자는 정확한 수학적 최적해보다 해석 가능한 편성안을 원한다.
```

### 14.3 점수는 정렬용 내부 지표

집단 점수와 추천 점수는 인게임 공식이 아니다.

목적:

```text
여러 후보를 비교하기 쉽게 정렬
좋은 집단을 상단에 표시
연결 밀도와 능력치를 함께 반영
```

### 14.4 결혼 후보 제외

결혼 후보와 미혼 가정 시뮬레이션은 현재 출력하지 않는다.
결혼은 기본 편성 점수에도 반영하지 않는다.
